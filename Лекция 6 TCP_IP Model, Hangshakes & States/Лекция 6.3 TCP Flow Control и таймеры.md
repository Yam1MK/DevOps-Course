# TCP Flow и Congestion Control

### Flow Control

- <ins>Защищает буфер</ins> принимающей ОС от переполнения (Receive Window)

- Отправитель ограничен In-flight данными из rwnd

- При чтении данных приложением ядро шлет Window Update
<br>
<br>
<br>
### Zero Window

- Возникает, если приложение (L7)  <ins>полностью зависло и не читает сокет</ins>

- В таком случае ядро анонсирует Window Size = 0

- Отправитель полностью прекращает трансляцию, но соединение (ESTABLISHED) не рвется
<br>
<br>
<br>
### Nagle

Блокирует **Silly Window Syndrome** (когда данные шлются по 1 байту) на стороне отправителя

Буферизирует мелкие пакеты, если в сети уже есть In-flight данные

пакет уходит только при получении ACK или накоплении MSS
<br>
<br>
<br>
### TCP_NODELAY и TCP_CORK

Алгоритм Нейгла создает проблемы для игр (FPS) и HFT-трейдинга

*TCP_NODELAY* принудительно отключает Нейгла (мгновенная отправка)

*TCP_CORK* жестко "затыкает" сокет, накапливая данные до ручного снятия

Оптимизация HTTP. TCP_CORK позволяет Nginx склеить HTTP-заголовки и файл (из sendfile) в памяти ядра
<br>
<br>
<br>
### Congestion Control (cwnd) vs Flow Control (rwnd)

- Размер окна передачи (W) вычисляется как min (cwnd, rwnd)

- Значение cwnd никогда не передается в заголовке, оно вычисляется скрыто
<br>
<br>
<br>
### Принцип AIMD (Reno)

Additive Increase (AI): <ins>линейный рост cwnd на 1 MSS за каждый RTT</ins>

Multiplicative Decrease (MD): резкая срезка окна <ins>вдвое при потере</ins>

Создает "пилообразный" (sawtooth) профиль утилизации канала
<br>
<br>
<br>
### TCP BBR

**BBR** <ins>игнорирует потерю пакета</ins> как первичный сигнал перегрузки

Непрерывно замедляет Bottleneck Bandwidth (BtlBw) и минимальный RTT

Держит в полете данных ровно на объем BDR
<br>
<br>
<br>
### Проблема Bufferbloat и AQM

Loss-based TCP (Reno, CUBIC) заполняет буфер роутера до отказа (Tail Drop)

Огромные буферы генерируют феномен **Bufferbloat** - <ins>мега-задержки без потерь</ins>

Требуются механизмы **AQM** (например, CoDel) на стороне провайдеров
<br>
<br>
<br>
### BBR FSM

Отказ от всплесков: пакеты <ins>жестко распараллеливаются во времени</ins> (Pacing)

Фаза ProbeBW: цикличная игра множителем (1.25 и 0.75) для оценки канала

Фаза ProbeRTT: срезка окна раз в 10с для оценки чистого пинга

Взаимоисключение метрик. Нельзя изменить макс-пропускную способность и мин-задержку одновременно; цикл гарантирует актуальность модели сети
