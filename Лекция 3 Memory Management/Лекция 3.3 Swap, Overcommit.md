# Swap, Overcommit
#### Концепция подкачки (Swap)

Расширение горизонтов RAM. Swap - это выделенный раздел на диске, куда ядро выгружает (page-out) "холодные" анонимные страницы, чтобы освободить место в физической RAM. 

Парадокс производительности. Использование медленного диска может ускорить систему в целом. Выгрузив в Swap мертвый код (например, функции инициализации программы), ядро освобождает быструю RAM для горячего файлового кэша.

Команды *swapon --show* для проверки статуса и *swapoff -a* (с последующим swapon -a) для принудительного переноса данных из Swap обратно в RAM (если памяти достаточно).


#### Исторический миф "Swap - это зло"

Наследие эпохи жестких дисков (HDD). Механические диски имели катастрофически низкий IOPS. Чтение из Swap парализовывало сервер (Thrashing), превращая его в "тыкву".

Смена парадигмы с приходом NVMe. Современные твердотельные накопители обеспечивают сотни тысяч IOPS с микросекундными задержками. Swap на NVMe работает быстрее, чем доступ к некоторым удаленным NUMA-узлам, делая подкачку легитимным буфером безопасности.

Опасность серверов без Swap. Отключение Swap лишает ядро маневра - анонимную память вытеснить некуда, ядру приходится агрессивно сбрасывать весь кэш (Page cache), убивая I/O производительность (доводя систему до OOM).


#### Параметр vm.swappiness

Настройка эвристики вытеснения. Этот параметр (от 0 до 100) задает баланс для kswapd: что вытеснять при нехватке памяти - скидывать анонимную память в Swap или удалять файловый кэш. Это не порог срабатывания, а именно "весовой коэффициент" предпочтения.

Стратегии тюнинга. Высокое значение заставит ядро агрессивно сбрасывать память процессов в Swap, защищая файловый кэш. Низкое значение (1-10) предписывает ядру жертвовать кэшем файлов до самого конца.

В Ubuntu дефолтное значение 60 подходит для десктопов, но вредно для СУБД.


#### Динамическое сжатие RAM: ZRAM

Виртуальный Swap в RAM. ZRAM создает блочное устройство прямо в RAM, которое сжимает помещаемые в него вытесненные страницы (обычно алгоритмами zstd или lz4).

Распаковка страницы алгоритмом zstd занимает микросекунды - это на порядки быстрее чтения с самого быстрого NVMe диска. Нагрузка на современные многоядерные CPU при этом практически незаметна.

Защита твердотельных накопителей. В отличие от классического Swap-файла, ZRAM не производит физических циклов записи на SSD. Это продлевает срок службы (TBW) накопителей на серверах с интенсивной подкачкой.


#### Swap Thrashing

Смертельная спираль подкачки. Состояние системы, когда она тратит 99% процессорного времени на перенос страниц между RAM и Swap- диском, а не на выполнение полезного кода приложения. Главный риск неправильно настроенного Swap - сервер перестает отвечать на сетевые запросы.

Причины возникновения. Возникает, когда рабочий набор (Working Set) активной памяти приложения физически больше, чем доступная RAM. Ядро выгружает страницу, но она немедленно требуется приложению снова (Major Page Fault).


#### vm.min_free_kbytes

Неприкосновенный запас ядра. Этот параметр *sysctl* резервирует абсолютный минимум свободной физ. памяти, которую не отдадут ни Page Cache, ни пользовательским процессам. Это необходимо для работы прерываний и сетевых драйверов.

Предотвращение дедлоков. Если свободная память упадет до 0, а сетевому драйверу понадобится буфер для принятия пакета (который скажет ядру освободить память) - система повиснет. Резерв гарантирует, что у драйверов всегда есть память.

Правила настройки. Дефолтные значения слишком малы для серверов с десятками гигабайт RAM. Рекомендуется устанавливать этот параметр в диапазоне 2-3% от общего объема RAM (но не более 1-2 ГБ) для стабильной работы kswapd.


#### Memory Overcommit

Эвристическая раздача ресурсов. По умолчанию (значение 0) ядро Linux позволяет процессам суммарно выделять (через malloc) больше виртуального адресного пространства, чем физически существует в RAM и Swap. Основано на факте ленивой аллокации - программы редко пишут во всю выделенную память.

Эффективность. Повышает утилизацию дорогостоящих серверов, позволяя разместить больше виртуальных машин или контейнеров. Это фундамент экономии всех облачных провайдеров.

Риск каскадного отказа. Если одновременно решат записать в свои "бумажные" адреса (вызвав Page Faults), физической RAM не хватит, и  запустится массовая резня OOM.


#### Режим vm.overcomit_memory = 2

Strict Overcommit. Перевод ядра в режим, где оно физически отказывает процессам в выдаче виртуальных адресов, если исчерпан лимит. Запрос malloc() вернет указатель NULL. Система теряет в гибкости, но приобретает 100% стабильность.

Формула расчета лимита. Максимально доступная для процессов память высчитывается строго: Swap + (RAM * vm.overcommit_ratio / 100). Ни один байт сверх этой формулы выдан не будет, OOM Killer остается без работы.

Этот режим противопоказан для десктопов, но обязателен для определенных инфраструктур. Обычно, настраивают этот режим исключительно на выделенных серверах для критичных баз данных (Redis, PostgreSQL), чтобы избежать их убийства.


#### Утечка памяти (Memory Leaks)

Классическая утечка (в C/C++) - это логическая ошибка, когда процесс выделил память, но не вызвал free() перед потерей указателя на нее. Память всегда исключается из пула, медленно, но верно подводя систему к OOM.

Сложность диагностики. Утечки развиваются медленно и маскируются под легитимный рост нагрузки. На дашбордах это выглядит как медленно ползущий вверх график потребления (sawtooth, pattern), который лечится только перезапуском.

Пользовательские инструменты. Для отладки бинарников применяются тяжеловесные профилировщики. Например, запускают программу через *valgrind --leak-check=full./app*, который перехватывает все аллокации, ценой 10-кратного замедления работы.
