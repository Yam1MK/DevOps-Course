# Кэширование, Slab-аллокатор и вытеснение страниц
#### Свободная память - потраченная впустую память

Ядро стремится утилизировать 100% физической RAM, заполняя свободное место кэшами дисковых операций. Это нормальное, задуманное архитекторами поведение.

RAM выдает данные за наносекунды, а NVMe диски - за микросекунды. Кэширование жизненно необходимо для компенсации этой разницы.

Правильное чтение утилиты free. Оценка здоровья системы. Пример: в выводе *free -h* в Ubuntu смотреть нужно не на столбец free, а **на столбец available** - это память, которую ядро готово мгновенно отдать приложениям, сбросив кэши.

#### Page Cache

Прозрачный посредник ввода-вывода. Все системные вызовы read() и write() по умолчанию проходят через Page Cache в RAM. Приложение думает, что читает с диска, но ядро отдает файл из RAM.

Если два процесса читают один файл логов, файл загружается в кэш только один раз. Это обеспечивает невероятную скорость работы веб-серверов при раздаче статического контента.

SRE могут очистить кэш для проведения чистых бенчмарков. Команда *echo 1 > /proc/sys/vm/drop_caches* мгновенно инвалидирует весь чистый Page Cache в системе.

#### Clean, Dirty Pages

**Clean Pages** - данные в памяти идентичны данным на дисковом накопителе. Их можно безопасно и мгновенно удалить (вытеснить) при нехватке памяти, система ничего не потеряет.

**Dirty Pages** - приложение изменило файл памяти, но ядро ещё не записало эти изменения на физ. диск. Агрегация мелких записей в RAM снижает износ SSD и повышает пропускную способность, но создает риск потери данных при сбое питания.

Мониторинг риска данных. SRE важно знать, сколько данных "висит" в памяти. *cat /proc/meminfo | grep Dirty* показывает текущий объем несинхронизированных с диском байт.

#### Writeback (Отложенная запись)

Ядро запускает фоновые потоки (flush threads), которые периодически сканируют память и сбрасывают грязные страницы на диск. Это обеспечивает баланс между производительностью и консистентностью данных.

Если грязных страниц становится слишком много, ядро блокирует процессы, заставляя записать страницы на диск. Это защищает систему от исчерпания RAM, но вызывает резкие спайки задержек у приложения.

Тюнинг параметров *sysctl*. SRE управляет агрессивностью сброса. Параметры *vm.dirty_background_ratio* (порог старта фонового сброса) и *vm.dirty_ratio* (порог жесткой блокировки процессов) настраивается в зависимости от скорости дисковой подсистемы.

#### Page Cache - fsync и msync

Системные вызовы ,заставляющие ядро немедленно сбросить грязные страницы конкретного файла в энергонезависимую память. Гарантирует консистентность - системный вызов блокируется, пока диск не подтвердит физ. запись.

ACID в базах данных. Надежность реляционных СУБД полностью строится на *fsync()* при записи в журналы транзакций (WAL/Redo log). Только так БД гарантирует, что подтвержденная транзакция переживет отключение питания.

Трассировка медленных синхронизаций. Если БД "тормозит", виноват может быть медленный диск/

#### vmtouch

Утилита, показывающая, какая часть конкретного файла или директории в данный момент находится в Page Cache. Позволяет точно узнать, загружен ли индекс БД в быструю RAM.

Принудительный "прогрев" кэша. Перед подачей рабочего трафика на реплику БД её память необходимо заполнить горячими данными. Предотвращает холодный старт и медленные ответы пользователям при первом обращении.

Использование в Ubuntu. Команда *vmtouch -t* жестко прочитывает файл в память, а флаг *-e* - принудительно вытеснит его из RAM, освободив место.

#### Директивы madvise / fadvise

Подсказки ядру от программиста. Системные вызовы, позволяющие приложению заявить о своих намерениях: "буду читать последовательно" или "данные больше не понадобятся". Эвристики ядра не идеальны - подсказки помогают оптимизировать опережающее чтение (readahead).

Защита кэша от вымывания. При стриминге огромного файла он может вытеснить из Page Cache все полезные индексы. Использование флага *FADV_DONTNEED* говорит ядру мгновенно удалять прочитанные страницы бэкапа из кэша.

Практическое применение в скриптах. SRE могут применять эту логику без изменения кода БД. Утилита nocache перехватывает вызовы и подставляет *FADV_DONTNEED*. Запуск *nocache tar -czf backup.tar /data* создаст архив, не засоряя RAM.

#### Алгоритмы замещения страниц (Page Reclaim)

Механизм выживания ОС. Логика ядра, определяющая, какие именно страницы памяти следует удалить (кэш) или выгрузить в Swap (анонимная память), когда свободная RAM заканчивается. Жизненно важный процесс для предотвращения ООМ.

Разделение горячего и холодного. Главная задача алгоритма - предсказать будущее: сохранить в RAM те страницы, к которым скоро обратятся, и выбросить "холодные" (давно не используемые). Неверный выбор приводит к постоянному чтению одних и тех же данных с диска.

Демон *kswapd*. Фоновый процесс ядра, отвечающий за асинхронную очистку. Процесс *kswapd0* активируется автоматически, как только объем свободной памяти в Ubuntu падает ниже порога *vm.min_free_kbytes*.

#### Active-Inactive LRU

Исторический стандарт Linux. Десятилетиями ядро делило память на два списка: Active (активно используемая) и Inactive (кандидаты на удаление). Простая реализация алгоритма LRU (Least Recently Used) с использованием "бита доступа" процессора.

Архитектурные ограничения. На современных серверах со 100+ ГБ RAM этот алгоритм деградирует, требуя постоянного линейного сканирования огромных списков (Page Scanning). Это приводит к тому, что демон kswapd начинает потреблять 100% ядра CPU просто пытаясь найти память.

Опасность Trashing'a. Алгоритм плохо справляется с потоковым чтением больших файлов (например, бэкапов), вымывая из списка Active действительно полезные данные БД, что вызывало жестокие "тормоза".

#### MGLRU - "Часы с двумя стрелками"

Новое поколение вытеснения (Multi-Gen LRU). MGLRU делит память не на 2 списка, а на динамическое количество поколений (generations) в зависимости от частоты доступа. Меняет подход - вместо сканирования неактивных страниц, MGLRU отслеживает активные, снижая объем работы.

Механика двух стрелок. "Стрелка старения" сканирует биты доступа и переносит страницы в молодое поколение, а "стрелка вытеснения" собирает страницы из старейшего поколения. Элегантная абстракция, минимизирующая блокировки (lock contention) в ядре при поиске памяти.

Внедрение MGLRU радикально снижает загрузку CPU при нехватке памяти (до 40-50%) и предотвращает зависания при резких скачках аллокации (спайках трафика).
