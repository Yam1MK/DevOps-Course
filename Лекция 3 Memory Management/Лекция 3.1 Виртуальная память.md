# Виртуальная память
#### Виртуальная память

Ядро как слой абстракции ресурсов. ОС берез на себя всю работу по распределению физической оперативной памяти (RAM).

Строгая изоляция адресных пространств. Каждый процесс работает в собственной виртуальной "песочнице". Ошибка или злой умысел в одном приложении не позволит ему прочитать пароли из памяти соседнего процесса.

Справедливый арбитраж под нагрузкой. Ядро динамически решает ,у кого забрать память, а кому выдать.

Процессу удобнее работать с непрерывной памятью, в том числе и для безопасности. Для этого используется виртуальная память.
<br>
<br>
#### Физическая память

Иерархия аппаратной памяти. Данные движутся от медленных дисков в RAM, затем в кэши L3/L2/L1 и регистры процессора. RAM служит критическим мостом, без нее процессор простаивал бы миллионы тактов в ожидании данных с диска.

Физическая адресация и контроллер памяти. Аппаратная память представляет собой сплошной массив байтов с физическими адресами. Прямая работа с физ. адресами опасна из-за фрагментации - сложно найти непрерывный кусок памяти для большой программы.

**Аппаратная** интроспекция в Linux. Инженеры должны понимать аппаратную топологию сервера перед программным тюнингом. Утилита *dmidecode -t memory* позволяет увидеть физические слоты и тип DDR.
<br>
<br>
#### Концепция виртуальной памяти

Иллюзия бесконечной памяти. Каждому процессу кажется, что он один владеет терабайтами непрерывной памяти, начинающейся с нулевого адреса. Это упрощает компиляцию программ.

Решение проблемы внешней фрагментации. Логически непрерывный виртуальный буфер может быть физически разбросан по разным микросхемам RAM. Ядро "собирает" этот пазл прозрачно для приложения, утилизируя каждый свободный байт.

В Linux виртуальное пространство делится на сегменты (текст, данные, хип, стек). Инженер может прочитать файл */proc/\<pid>/maps*, чтобы увидеть точную раскладку виртуальных адресов любого запущенного демона.
<br>
<br>
#### Страничная организация (Paging)

Квантование памяти на страницы. Ядро делит виртуальную память на "страницы" (Pages), а физическую - на "кадры" (Frames) фиксированного размера. Управление блоками (обычно по 4 кб) намного эффективнее, чем отслеживание каждого отдельного байта.

Проверка параметров системы. Размер страницы жестко зашит в архитектуру, но его нужно знать для расчётов лимитов. Команда *getconf PAGE_SIZE* в Ubuntu покажет базовый размер (4096 байт для x86_64 архитектуры).
<br>
<br>
#### Таблицы страниц (Page Tables)

Словари трансляции адресов. Ядро хранит маппинг - какая виртуальная страница каждому физическому кадру соответствует. Без этой таблицы процессор не знал бы, куда физически обращаться при инструкции MOV.

Многоуровневая древовидная структура. Вместо огромного плоского списка используется 4- или 5- уровневые деревья. Придумано для экономии памяти самого ядра. Плоская таблица занимала бы ГБ RAM для каждого процесса, а дерево позволяет хранить пустые узлы.

Мониторинг накладных расходов. Сами таблицы тоже потребляют память, что становится проблемой на серверах с тысячами процессов. Объем памяти, съеденный таблицами, можно отследить метрикой PageTables в */proc/meminfo*.
<br>
<br>
#### TLB - Translation Lookaside Buffer

Аппаратный кэш внутри процессора. TLB хранит последние результаты перевода виртуальных адресов в физические. Обход 4 уровней таблиц в RAM при каждой инструкции замедлил бы CPU в десятки раз. TLB выдает ответ мгновенно.

Проблема промахов (TLB Miss). Если адреса нет в TLB, контроллер памяти (MMU) вынужден делать медленный обход таблиц (Page Walk). Для баз данных с хаотичным доступом к огромным массивам памяти TLB Miss становится узким местом производительности.

Профилирование промахов в Linux. Инженер может измерить эту архитектурную метрику на работающем приложении. Утилита *perf stat-e dTLB-load-misses \<command>* покажет, насколько сильно процесс страдает от нехватки объёма TLB.
<br>
<br>
#### HugePages

Увеличение размера кванта памяти. Использование страниц размером 2 МБ или 1 ГБ вместо стандартных 4 КБ. Одна запись в TLB теперь покрывает в 500 раз больше памяти, что снижает вероятность TLB Miss для баз данных.

Отсутствие вытеснения в Swap. Память, выделенная через стандартные HugePages, блокируется в RAM и никогда не уходит в SWAP.

Настройка и мониторинг в Ubuntu. SRE выделяет этот пул при загрузке системы для избежания дефрагментации. Настройка через *sysctl vm.nr_hugepages=1024*. Контроль утилизации: *cat /proc/meminfo | greb Huge*.
<br>
<br>
#### Аллокация памяти в user-space

Разделение зон ответственности. Ядро оперирует только целыми страницами, а приложению нужны объемы по 10-20 байт. Стандартная библиотека (glibc malloc) запрашивает у ядра большие блоки, а затем сама "нарезает" их мелкими кусками для нужд программы.

Системные вызовы brk() и mmap(). brk() сдвигает кучи (heap), а mmap() создает новые независимые регионы памяти. mmap придуман для безопасного выделения больших объемов, которые потом можно вернуть ядру (munmap), не дожидаясь освобождения всей кучи.

Трассировка аллокаций. SRE может заглянуть "под капот" любого бинарника, чтобы увидеть, как он просит память. Использование *strace -e trace=memory -p \<PID>* покажет все системные вызовы выделения в реальном времени.
<br>
<br>
#### Demand Paging

Память выдается только на бумаге. при вызове malloc() ядро резервирует виртуальные адреса, но не выделяет ни одного байта физической RAM. Ядро "не доверяет" программистам, так как программы часто запрашивают память про запас.

Экономия физических ресурсов. Физический кадр подключается только в момент первой попытки записи в эту память. Это позволяет запускать приложения мгновенно и поддерживать механизм Overcommit (переподписки памяти)/

Разница между VIRT и RES. В утилите top столбец VIRT показывает, сколько памяти процесс "попросил", а RES (Resident) - сколько физической RAM фактически выдало.
<br>
<br>
#### Page Fault

Аппаратное прерывание от CPU, возникающее, когда процесс обращается к виртуальному адресу без привязки к физической RAM. Это не ошибка приложения, а штатный механизм сигнализации ядру.

Ядро приостанавливает поток, находит физ. страницу, заполняет ее нулями (для безопасности), прописывает маппинг и возобновляет поток.

Высокая частота прерываний может указать на неоптимальную работу с памятью. Использование утилиты pidstat -r 1 позволяет отслеживать интенсивность генерации фолтов конкретным приложением.

#### Minor vs Major Page Faults

- **Minor Fault** - страница была успешно найдена в свободной RAM или page кэше. Обрабатывается ядром за микросекунды, это нормальное поведение при ленивой аллокации.

- **Major Fault** - случай, когда процесс не может получить память из RAM, и ядру приходится обращаться к жесткому диску для копирования (из swap или файла). Вызывает миллисекундные блокировки потока (I/O Wait), значительно снижая отзывчивость приложения.

Поиск узких мест. SRE должен реагировать на появление Major Faults как на симптом деградации (thrashing). Команда *ps -eo min_flt,maj_flt,cmd* покажет накопительную статистику процесса, где высокое значение maj_flt указывает на проблемы.
<br>
<br>
#### Системный вызов fork() и память

Быстрое клонирование процессов. В Linux fork() не копирует всю физическую память родителя в дочерний процесс. Копируются исключительно структуры таблиц страниц.

Совместное владение (Sharing). Сразу после форка оба процесса указывают на одни и те же физ. кадры в RAM. Это обеспечивает колоссальную экономию памяти для многопроцессорных архитектур, разделяющих общие структуры данных.

Интроспекция деревьев процессов. Для понимания архитектуры форков SRE используют визуальные инструменты. Утилита htop (по клавише F5) выстраивает иерархическое дерево, позволяя увидеть связь между родительским демоном и его пулом воркеров.
<br>
<br>
#### Механизм Copy-on-Write (CoW)

Отложенное копирование при записи. При форке все разделяемые страницы помечаются ядром как "только для чтения" (Read Only). Как только один из процессов пытается изменить переменную, возникает Page Fault. Ядро перехватывает его, копирует страницу в новое место физической RAM и разрешает запись.

Бесконечное разделение памяти. Если память только читается, она остается разделяемой навсегда. Это суть паттерна "pre-forking" - загрузить тяжелые данные в родителя до форка, чтобы дети читали их бесплатно по памяти.

Честный учет памяти (Метрика PSS). Обычные утилиты завышают потребление форков, считая CoW память каждому процессу. Метрика PSS (Proportional Set Size) в */proc/\<pid>/smaps* делит объем общей памяти на число владельцев, показывая реальный след процесса в ОС.
<br>
<br>
#### Типы памяти

- **Анонимная память** - память без файловой основы. Это динамическая память процесса, которая не имеет физического файла на диске. Называется "анонимной", потому что ядро не знает, с каким файлом ассоциировать эти временные данные (стек вызовов, переменные в куче malloc). 

- **Файловая память (File-backed Memory)** - память, спроецированная из файлов. Это бинарные коды программ, разделяемые библиотеки и файлы, прочитанные через mmap. 
<br>
<br>
#### Архитектура NUMA (Non-Uniform Memory Access)

Локализация банков памяти. В современных многосокетных серверах слоты оперативной памяти аппаратно привязаны к конкретному физ. процессору. Это придумано для обхода узкого места системной шины, позволяя CPU обращаться к "своей" памяти максимально быстро.

Неравномерный доступ. Процессор может читать память соседнего узла по интерконнекту (QPI/UPI), но это происходит на 20-30% медленнее. SRE должен следить, чтобы потоки тяжелых баз данных работали с локальной для их CPU памятью, избегая кросс-узлового трафика.

Инспекция топологии. Настройка баз данных невозможна без понимания физ. архитектуры сервера. Установка пакета *numactl* в Ubuntu и запуск *numactl --hardware* покажет распределение ядер и RAM по узлам, а также матрицу задержек между ними.
<br>
<br>
#### CPU Pinning и Memory Affinity

Жесткая изоляция ресурсов. SRE могут явно указать ядру запретить планировщику перемещать процесс и обязать его использовать только конкретный банк RAM. Придумано для достижения максимальной предсказуемости (Determinism) в Real-Time системах и NFV (Network Function Virtualisation).

Настройка политик (Interleave). Политика interleave заставляет ядро распределять страницы процесса равномерно по всем NUMA-узлам. Это стандартная рекомендация для In-Memory СУБД (Redis, Aerospike), так как балансирует пропускную способность шины памяти.

Запуск с привязкой. SRE управляет этими политиками без изменения кода приложения. В Ubuntu запуск процесса через обертку *numactl --interleave=all \<deamon>* меняет логику выделения памяти для этого демона.
<br>
<br>
#### Утилиты SRE - ps, top, htop

В top столбец SHR (Shared) указывает на объем памяти, потенциально разделяемой с другими процессами (библиотеки, tmpfs). Понимание разницы VIRT, RES, SHR защищает SRE от ложных выводов о том, что процесс "утекает", когда он просто загрузил .so библиотеку.
<br>
<br>
#### Утилиты SRE - pmap, smaps

Картирование памяти (Memory Map). Детальная разбивка того, на что именно процесс тратит память (размер кучи, стека). top показывает только общую сумму, а pmap позволяет локализовать сегмент, в котором происходит утечка.

Грязные страницы (Dirty Pages). */proc/\<pid>/smaps* показывает статус каждой страницы - была ли она модифицирована процессом. Поиск аномально большого пула грязных страниц помогает выявить проблемы с отложенной записью (Writeback) на диск.

Автоматизация расчетов. Парсинг smaps - классическая задача для Bash-скриптов. *cat /proc/\<pid>/smaps | grep -i pss | awk '{Total+=$2} END {print Total/1024"MB"}'* быстро подсчитывает точное (PSS) потребление памяти конкретным PID.
