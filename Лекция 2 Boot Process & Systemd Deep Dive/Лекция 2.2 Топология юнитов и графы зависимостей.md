#### Иерархия директорий systemd

- */usr/lib/systemd/system/* - файлы апстрима (поставляемые через apt) 
- */run/systemd/system/* - эфемерные юниты, генерируемые в реальном времени (не трогаем)
- */etc/systemd/system/* - пользовательские настройки администратора (SRE)
- Строгий приоритет: файлы в */etc* полностью перекрывают файлы в */usr*
- Изменение файлов в */usr/lib* вручную **категорически запрещено**

#### Механизм Drop-in файлов (override.conf)

- Служит для **точечного изменения** одной директивы (например, лимита памяти)
- Создается в директории *<имя_юнита>.service.d/*
- Утилита *systemctl edit \<unit>* автоматизирует создание этих файлов
- Сохраняет базовый юнит вендора нетронутым для получения патчей безопасности
- Применяется поверх основного файла конфигурации на лету

#### Основные типы юнитов (.sevice и .target)

- *.service*: **Управляет** демонами. Определяет ExecStart, Restart, Type
- *Type=simple*: считается **запущенным** сразу после выполнения команды
- *Type=notify*: **Ожидает** сигнала готовности через D-Bus
- *.target*: логическая **точка синхронизации**, замена старых runlevels
- *multi-user.target*: Эквивалент классического **консольного режима**

#### Типы юнитов (.socket и .timer)

- *.socket*: Описывает **сетевой или UNIX-сокет.** Используется для ленивой загрузки
- *.timer*: Замена демона cron с **миллисекундной** точностью
- Таймеры предотвращают наложение задач (overlapping), если предыдущая ещё выполняется
- Поддержка монотонного времени (OnBootSec=) и календарного (OnCalendar=)

#### Socket Activation

- Systemd **заранее создает** прослушивающий порт до запуска самого демона
- При поступлении пакета ядро **буферизирует** его, а systemd запускает нужный сервис
- Файловый дескриптор сокета (FD) бесшовно передается приложению
- Если сервис простаивает, он может быть **выгружен** из ОЗУ
- Архитектура заимствована и улучшена из классического демона *inetd*

#### Разрешение Race Conditions через сокеты

- Классическая проблема: микро сервис А **падает**, потому что база данных Б **еще грузится**
- Решение через сокеты: оба сервиса абсолютно **одновременно**
- Сервис А отправляет запрос к базе Б
- Поскольку сокет базы Б уже существует, ядро просто приостанавливает запрос сервиса А
- Как только база Б инициализируется, она мгновенно обрабатывает ожидающий запрос

#### Зависимости доступности (Wants против Requires)
- Обе директивы указывают сис. менеджеру запустить связанные юниты
- *Requires=*: Строгая (жесткая) зависимость. Падение дочернего сервиса **убьет** родительский
- *Wants=*: Мягкая зависимость. Если дочерний сервис не запустился, родитель **продолжит** работу
- В SRE-практиках *Wants=* является **стандартом** по умолчанию

#### Зависимости порядка (Before против After)

- *Requires* и *Wants* **НЕ управляют порядком** старта во времени
- По умолчанию systemd запускает зависимые юниты **параллельно**
- *After=*: Не запускать текущий сервис, пока указанный не перейдет в состояние **active**
- *Before=*: Запустить текущий сервис **до старта** указанного
- Комбинация Wants=X и After=X - самый **частый паттерн** в \[Unit]

#### Неявные зависимости (Implicit Dependencies)

- Менеджер systemd **автоматически добавляет** связи, анализируя тип юнита
- Пример 1: Юниты с *Type=dbus* неявно получают *Requires=dbus.socket*
- Пример 2: Юниты .*mount* автоматически получают зависимости от файлов устройств (*.device*)
- Снижает размер конфигов (принцип DRY)
- Позволяет избежать тривиальных синтаксических ошибок

#### Генераторы конфигов (Generators)

- Бинарные исполняемые файлы в */usr/lib/systemd/system-generators/*
- Отрабатывают на **сверхраннем** этапе загрузки до монтирования корневой ФС в rw
- Динамически транслируют legacy-файлы в нативные юниты
- Пример: *systemd-fstab-generator* превращает */etc/fstab* в *.mount* юниты
- Результат их работы складывается в эфемерную директорию */run/systemd/generator/*

#### Шаблонизация юнитов (Template Units)

- Имя файла содержит символ *@,* например *openvpn@.service*
- позволяет **запускать множество независимых инстансов** (процессов) из одного файла
- Аргумент после @ передается внутрь юнит-файла как переменная %i
- *systemctl start openvpn@client1* создаст инстанс со значением client1
- Идеально для **масштабирования** контейнеров, сетевых интерфейсов и туннелей

#### Разница между Daemon Restart и Reload

- *Restart* отправляет сигналы *SIGTERM*, а затем *SIGKILL*. Процесс **полностью пересоздается**
- При Restart сбрасываются все текущие сетевые сессии клиентов
- *Reload* (команда *systemctl reload*) отправляет сигнал *SIGHUP*
- Процесс **перечитывает** свои **конфиги** (например, nginx.conf) "на горячую" - клиентский трафик не прерывается

#### Состояния юнитов

1) *Loaded*: Файл **успешно прочитан** в RAM
2) *Active (running)*: Главный процесс PID **выполняется**
3) *Active (exited)*: Скрипт успешно отработал и **завершился** (характерно для Type=oneshot)
4) *Disabled*: Юнит **не добавлен в автозагрузку** (нет ссылки в .wants)
5) *Masked*: Запуск жестко **заблокирован** (файл симлинком указывает на /dev/null)
