# HTTP/3 и протокол QUIC

### QUIC - Транспорт нового поколения (RFC 9000)

Работает поверх UDP (User Datagram Protocol) для обхода TCP Ossification

Переносит управление потоками, криптографию (TLS 1.3) и контроль перегрузки в <ins>пользовательское пространство (User-space)</ins> - не нужно обновлять ядро ОС

<ins>Шифрует практически весь заголовок</ins>, ослепляя Middleboxes и предотвращая их вмешательство

QUIC был создан поверх UDP как <ins>единственный рабочий способ обойти "окостенение" интернета и перенести всю логику управления транспортом в пользовательское пространство (User-space) браузера</ins>
<br>
<br>
<br>
### Устранение TCP HoL Blocking

В QUIC **мультиплексирование потоков** реализовано на транспортном уровне

Потеря UDP-дейтаграммы задерживает только те QUIC-потоки, чьи фреймы STREAM находились в утерянном пакете

Остальные потоки продолжают читаться приложением из буфера приема без пауз

Теперь <ins>потеря одного UDP-пакета влияет только на конкретный поврежденный поток данных</ins>, позволяя остальным потокам беспрепятственно загружаться
<br>
<br>
<br>
### Connection ID - CID

Отказ от 4-Tuple (Source IP, Source Port, Dst IP, Dst Port) для идентификации соединения

Внедрение 64-битного (или более) криптографического **CID** в заголовок QUIC пакета

Позволяет серверу <ins>маршрутизировать пакеты к правильному сокету даже при изменении IP-адреса клиента</ins>

Придуман для отвязки криптографической сессии от физического IP-адреса, что решает многолетнюю проблему обрыва соединений при переходах через NAT и при смене сетей
<br>
<br>
<br>
### CID - Приватность и балансировка

В отличие от TCP, CID позволяет клиенту менять IP (миграция) без разрыва криптографического контекста

Для предотвращения трекинга перемещений клиента со стороны провайдеров (Linkability), <ins>CID регулярно ротируются</ins>

Балансировщики нагрузки используют встроенные в CID токены для маршрутизации UDP-пакетов на нужный бэкенд даже после смены IP. Это потребовало своих балансировщиков для QUIC

Регулярная ротация Connection ID была заложена в стандарт для того, чтобы <ins>провайдеры (ISP) и DPI не могли отслеживать физическое перемещение мобильного пользователя между базовыми станциями</ins>
<br>
<br>
<br>
### Интеграция TLS 1.3 в QUIC

<ins>QUIC не работает без шифрования (в отличие от TCP)</ins>. TLS 1.3 встроен на уровне протокола

QUIC инкапсулирует сообщения TLS (ClientHello) в специальные CRYPTO фреймы

QUIC использует ключи TLS (Secret Key Schedule) для защиты собственных пакетов на каждом этапе

Слияние транспортного и криптографического слоев было реализовано, чтобы установить полноценное защищенное соединение за 1 RTT (или 0 RTT), минуя долгие многоэтапные "рукопожатия" TCP+TLS
<br>
<br>
<br>
### Форматы пакетов QUIC

**Long Header** - используется <ins>до завершения рукопожатия</ins> (типы Initial, 0-RTT, Handshake, Retry)

**Short Header** - используется <ins>для пакетов 1-RTT (полезная нагрузка)</ins>, содержит минимальный оверхед

**Бит Spin (Spin Bit)** - опциональный бит в Short Header для <ins>пассивного измерения RTT</ins>

Внедрено для снижения накладных расходов: длинные заголовки с полными метаданными нужны протоколу только для установки связи, а короткие - для сверхбыстрой передачи полезной нагрузки
<br>
<br>
<br>
### Header Protection

В TCP Sequence Number передается в открытом виде (уязвимость к RST-injection)

QUIC <ins>шифрует номер пакета (Packet Number) и часть флагов</ins>, используя семпл (sample) из зашифрованной полезной нагрузки

Промежуточные узлы не могут вычислить объем переданных данных без ключей

Создана для защиты транспортных метаданных (Packet Number) от модификации и анализа транзитными узлами
<br>
<br>
<br>
### Header Protection: AES-CTR vs ChaCha20

Для защиты заголовков применяется алгоритм **AES-ECB** (в режиме счетчика) или **ChaCha20**

Алгоритм генерирует 5-байтную маску, которая применяется операцией XOR к байту флагов и номеру пакета

Это изолированный криптографический шаг, выполняемый после шифрования полезной нагрузки (AEAD payload protection)

Применение шифров поверх уже зашифрованной полезной нагрузки создает второй, независимый слой криптографии исключительно для <ins>сокрытия служебных полей транспортного заголовка</ins>
<br>
<br>
<br>
### Архитектура HTTP/3 (RFC 9114)

**HTTP/3** - это адаптация семантики HTTP к транспорту QUIC

<ins>Отказ от части фреймов HTTP/2</ins> (например, встроенного Flow Control), так как QUIC уже предоставляет их на уровне транспорта

Внедрение QPACK вместо HPACK для решения проблем синхронизации словарей

Необходимость перенести семантику HTTP поверх нового транспорта QUIC с удалением дублирующихся функций (например, управления потоком), которые теперь реализуются на транспортном уровне
<br>
<br>
<br>
### Cjngestion Control в QUIC

Перенос алгоритмов (NewReno, CUBIC, BBR) <ins>из ядра ОС в библиотеку (User-space)</ins>

QUIC номера пакетов строго монотонно возрастают (Monotonically Increasing), даже при ретрансмиссиях

Это решает проблему TCP Retransmission Ambiguity

Реализация алгоритмов в user-space позволила обновлять их вместе с браузером (например, внедрять новые версии BBR) мгновенно
<br>
<br>
<br>
### Amplification Attacks и защита

Из-за UDP злоумышленник может <ins>подделать Source IP и отправить мелкий пакет Initial, вызвав большой ответ (Amplification)</ins>

QUIC требует до подтверждения адреса клиента сервер не отправляет данных <ins>более чем в 3 раза превышающих размер запроса (3x Limit)</ins>

Использование пакета Retry с криптографическим токеном валидации адреса
<br>
<br>
<br>
### Конец эпохи TCP

UDP становится <ins>новым "IP-уровнем" для пользовательского пространства</ins>

IETF (Internet Engineering Task Forse) активно переводит DNS, SMB и SSH на транспорт QUIC

QUIC заставляет индустрию переосмыслить аппаратную архитектуру сетевых карт (NIC) для поддержки UDP Segmentation Offload

Окончательный переход индустрии на современную парадигму, где транспорт и криптография <ins>неразделимы и контролируются самим приложением</ins>, а не ядром ОС или сетевым транзитным оборудованием
