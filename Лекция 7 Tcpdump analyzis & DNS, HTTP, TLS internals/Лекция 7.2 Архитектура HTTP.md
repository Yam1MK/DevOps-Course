# Архитектура HTTP

### Ограничения HTTP/1.1

Текстовый stateless протокол с избыточными ASCII-заголовками (Header Bloat)

Механизм Keep-Alive позволяет переиспользовать TCP, но запросы обрабатываются строго последовательно

Pipelining (конвейеризация) потерпел крах из-за "окостенения" промежуточных прокси-серверов

Рост сложности веб-страниц (от одного HTML-документа к сотням ассетов, скриптов и картинок) привел к тому, что текстовая природа протокола и последовательная обработка запросов стали <ins>главным узким местом производительности Web 2.0</ins>

<br>

### Head-of-Line Blocking

В HTTP/1.1 сервер обязан отправлять ответы в <ins>строгом соответствии с порядком получения запросов</ins>

Если генерация ответа 1 занимает 5 секунд, ответы 2 и 3 блокируются в буфере

Workaround разработчиков: Domain Sharding (открытие 6 TCP-соединений к разным субдоменам)

Требование FIFO (First-In-First-Out) в HTTP/1.1 заставляло быстрые ответы (например, статику) простаивать в ожидании медленных (запросы к БД), катастрофически снижая утилизацию пропускной способности канала
<br>
<br>
<br>
### Переход к бинарному фреймированию в HTTP/2

Отказ от текстовых блоков: сообщение разбивается на <ins>фреймы с 9-байтным заголовком (Length, Type, Flags, Stream ID)</ins>

Парсинг бинарных данных <ins>требует меньше процессорного времени</ins> (CPU) по сравнению с поиском CRLF (\r\n)

Разделение данных (DATA frame) и метаданных (HEADERS frame)

Изобретено для исключения уязвимостей, связанных с ошибками парсинга текстовых границ (CRLF/HTTP Desync), снижения нагрузки на CPU и создания математически строгой основы для инкапсуляции и мультиплексирования
<br>
<br>
<br>
### Мультиплексирование потоков (Streams)

Stream ID (31 бит) идентифицирует <ins>независимый логический канал внутри одного TCP-соединения</ins>

Фреймы разных потоков перемешиваются (Interleaving): сервер может отправить чанк ответа 1, затем чанк ответа 2

<ins>Полное устранение Head-of-Line Blocking</ins> на уровне HTTP-приложения
<br>
<br>
<br>
### Flow Control в HTTP/2

Реализовано независимо от TCP на уровне каждого Stream и всего Connection

Использует фрейм WINDOW_UPDATE для выдачи "кредитов" на отправку данных сервером

Защищает клиент от <ins>переполнения буфера памяти при скачивании тяжелых медиафайлов</ins>

Разработано для предотвращения исчерпания оперативной памяти клиента (переполнения буферов) при загрузке множества тяжелых элементов одновременно по высокоскоростному мультиплексированному каналу
<br>
<br>
<br>
### HTTP/2 поверх TLS (h2)

- Спецификация допускает HTTP/2 без шифрования (h2c), но браузеры требуют TLS 1.2+ (h2)

- Выбор протокола происходит в рамках TLS-рукопожатия через расширение ALPN (Application-Layer Protocol Negotiation)

- HTTP/2 вводит черный список старых шифров TLS для обеспечения безопасности
<br>
<br>
<br>
### TCP Ossification

Промежуточное оборудование (Middleboxes) десятилетиями оптимизировалось под TCP-заголовки

Любая попытка внедрить новые флаги TCP приводит к <ins>отбрасыванию пакетов (Packet Drop)</ins>

Интернет "окостенел", сделав невозможной эволюцию протокола TCP

Мотивация создания (перехода на UDP): Агрессивная и некорректная оптимизация транзитного оборудования (NAT, DPI, Firewall) под формат TCP-заголовка сделала <ins>практически невозможным добавление новых флагов или изменение механики протокола</ins>
<br>
<br>
<br>
### Итог HTTP/2 и переход на QUIC

Бинарное фреймирование и HPACK успешно решили некоторые проблемы L7

TCP HoL Blocking делает HTTP/2 медленнее HTTP/1.1 в сетях с потерями пакетов > 2%

Вывод - для дальнейшего ускорения Web необходим <ins>отказ от TCP и создание нового транспорта</ins>

Индустрия пришла к консенсусу, что потенциал протокола HTTP/2 исчерпан, и дальнейшая оптимизация веба физически невозможна без <ins>полной замены самого транспортного уровня L4</ins>
