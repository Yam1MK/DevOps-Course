# Процессы в Linux

### Абстракция процесса в архитектуре ядра Linux

В Linux существует две конструкции: **процесс** (более высокоуровневая конструкция) и **поток** (единица исполнения данного процесса).

Ядро не делает строгих различий между "потоком" и "процессом" на уровне внутренних структур данных

Дизайн ядра Linux исторически избегал отдельных структур для потоков, предпочитая универсальную абстракцию "задачи" (tasks). Это упрощает логику планировщика.
<br>
<br>
<br>
### Идентификация задач - PID и TGID

С точки зрения user-space, каждый процесс уникально идентифицируется числовым значением **PID** (Process ID)

Значение PID остается неизменным на протяжении всей жизни процесса, после завершения он освобождает его для будущего переиспользования ядром.

Для поддержки многопоточности стандарта POSIX ядро использует концепцию групп потоков (Thread Groups), где все потоки разделяют общий **TGID** (Thread Group ID)
<br>
<br>
<br>
### Cache Locality и структуры ядра

Скорость работы центрального процессора многократно превышает пропускную способность шины оперативной памяти

Процессор загружает данные из основной памяти в L1/L2 кэш фиксированными блоками, называемыми строками кэша (**cache lines**)

Если важные и совместно используемые поля task_struct разбросаны по структуре, это приводит к неэффективному использованию кэша

Когда планировщик Linux сканирует структуры процессов, он извлекает из памяти строки кэша (обычно 64 байта)
<br>
<br>
<br>
### Выравнивание структур данных (Cache Alignment)

Поля внутри task_struct выравниваются по границам строк кэша (cache alignment), даже если это приводит к увеличению общего размера структуры (padding)

Невыровненные структуры теоретически экономят память, но заставляют процессор совершать доп. вычислительные циклы для склейки данных из разных строк кэша

Выровненные структуры (кратные степени 2) гарантируют консистентность и предсказуемость выборки данных из иерархии кэшей
<br>
<br>
<br>
### Системный вызов fork

Исторически системный вызов fork() создавал абсолютно идентичную копию родительского процесса, подразумевая их полную изоляцию

Эта операция требовала полного копирования всех сегментов данных, heap и стека из памяти родителя в память потомка

Разделяемым (read-only) оставался только сегмент машинного кода (text segment), что все равно делало операцию крайне медленной
<br>
<br>
<br>
### Оптимизация Copy-on_Write

Современный fork() в Linux опирается на механизм отложенного копирования - **Copy-on-Write**, который вместо копирования физ. страниц, дублирует только таблицы страниц (page tables), устанавливая для обоих процессов права Read-Only

Фактическое копирование физ. страницы происходит исключительно в момент возникновения исключения (page fault) при попытке записи в нее

Когда один из процессов пытается изменить переменную, процессор генерирует аппаратное прерывание (protection fault). Ядро перехватывает его, понимает, что страница является CoW-разделяемой, выделяет новый физ. фрейм, копирует туда данные, обновляет таблицу страниц виновника прерывания, давая ему права на запись, и возобновляет выполнение
<br>
<br>
<br>
### Системный вызов vfork

В качестве экстремальной оптимизации в системах BSD был внедрен вызов **vfork()**, при котором дочерний процесс выполняется <u>прямо в адресном пространстве родителя</u>, а выполнение родителя принудительно приостанавливается

Блокировка родителя необходима, так как параллельное выполнение двух потоков команд на одном неразделенном стеке немедленно привело бы к состоянию гонки и порче данных

В современном ядре Linux vfork() реализуется через установку специфических флагов состояния (включая TASK_KILLABLE), блокирующих родителя, но позволяющих прервать его в случае фатального зависания потомка
<br>
<br>
<br>
### Системный вызов clone

В современном ядре Linux системные вызовы fork и vfork фактически <u>являются алиасами</u> (псевдонимами) над универсальным вызовом **clone()**

clone() принимает набор битовых флагов (например, CLONE_VM, CLONE_FILES, CLONE_SIGHAND), определяющих, какие именно ресурсы будут разделены

Жесткая семантика традиционного fork() не позволяла развиваться сложным многопоточным архитектурам. Введение clone() предоставило разработчикам гранулярный контроль над ядром
<br>
<br>
<br>
### Взаимодействие процессов с Swap

Для предотвращения лишних дисковых операций ядро поддерживает структуру **Swap Cache** - список записей таблиц страниц для выгруженных физ. страниц

Swap Cache выступает в роли <u>буфера обмена</u>. Когда ядру необходимо освободить оперативную память, оно сначала проверяет Swap Cache. Если страница уже имеет валидную копию на диске и не была изменена, ядро может просто отбросить её из RAM без выполнения дорогостоящей операции записи на диск
<br>
<br>
<br>
### Swap Cache - обработка Page Fault при чтении

Если процесс пытается модифицировать страницу (возникает write access page fault), архитектура кэша подкачки меняет поведение

Ядро помечает страницу как "грязную" (dirty) и одновременно делает её доступной для записи в таблице страниц (writeable)

Соответствующая запись немедленно удаляется из Swap Cache, разрывая связь с копией на диске
<br>
<br>
<br>
### Базовые состояния процессов

Жизненный цикл процесса описывается конечным автоматом состояний, фиксируемых в task_struct

Состояние **R (Running/Runnable)** означает, что процесс либо <u>исполняется</u> на CPU, либо <u>готов к исполнению</u> и находится в очереди планировщика

Состояние **T (Stopped)** указывает на то, что процесс был <u>приостановлен</u> сигналом (например, SIGTPTP или SIGSTOP) для отладки или управления заданиями. Оно полезно для временного освобождения CPU от задачи без потери ее прогресса (CTRL + Z)

Разделение состояний необходимо планировщику для понимания того, какие задачи можно ставить на процессор
<br>
<br>
<br>
### Состояние S (Interruptible Sleep)

Подавляющее большинство фоновых процессов в системе находятся в состоянии **S (Interruptible Sleep)**, <u>ожидая завершения асинхронного события</u> (например, ввода с клавиатуры или пакета по сети)

Процесс в состоянии S может быть немедленно "разбужен" операционной системой при получении любого сигнала. Переход в состояние S выводит задачу из очереди планировщика

Ядро помещает дескриптор задачи в специальную очередь ожидания (wait queue), связанную с конкретным устройством или сокетом. Как только устройство сообщает о готовности данных через аппаратное прерывание, драйвер проходит по очереди ожидания и переводит процессы в R
<br>
<br>
<br>
### Состояние D (Uninterruptible Sleep)

Состояние **D (Uninterruptible Sleep)** представляет собой <u>сон, игнорирующий любые прерывания</u> пользовательского уровня

Процесс в состоянии D не может быть завершен даже сигналом SIGKILL, отправленным пользователем root

Основной триггер перехода в данное состояние  - ожидание завершения важных операций ввода-вывода или аппаратных блокировок

Процесс выйдет из состояния D только тогда, когда драйвер устройства сообщит об успешном завершении транзакции
<br>
<br>
<br>
### Планировщик задач CFS

Начиная с версии ядра 2.6 и до недавнего времени, стандартом де-факто являлся **Completely Fair Scheduler (CFS)**

CFS основывался на концепции виртуального времени (vruntime), стремясь предоставить каждой задаче строго пропорциональную долю процессора

Выбор следующей задачи для исполнения сводился к выбору узла с наименьшим значением vruntime ил красно-черного дерева (Red_Black tree)
<br>
<br>
<br>
### Ограничения CFS и потребность в EEVDF

Главным архитектурным недостатком CFS являлась его неспособность явно выражать требования к задержкам (latency constraints) для процессов

Для решения этой проблемы в ядро Linux 6.6 был интегрирован планировщик **EEVDF (Earliest Eligible Virtual Deadline First)**

EEVDF разделяет концепции "сколько времени положено процессу" и "как быстро он должен его получить", позволяя гибко управлять отзывчивостью без перекосов в общей утилизации CPU
<br>
<br>
<br>
### EEVDF - "Lag"

Основной метрикой планировщика EEVDF является показатель **"lag"** (отставание) для каждой задачи

Он вычисляется как разница между временем, которое процесс <u>должен был получить</u> по идеальной модели, и временем, которое он <u>фактически исполнил</u>

Процесс считается "подходящим" (eligible) для постановки на процессор только в том случае, если его lag больше или равен 0
<br>
<br>
<br>
### EEVDF - вычисление виртуальных дедлайнов

Среди всех "подходящих" (eligible) задач EEVDF должен выбрать ту, которая будет исполнена следующей

Для каждой задачи вычисляется "виртуальный дедлайн" путем сложения <u>времени ее допуска</u> (eligibility time) с <u>остатком выделенного кванта времени</u> (time slice)

Процессорное время отдается задаче с самым ранним (наименьшим) виртуальным дедлайном
<br>
<br>
<br>
### EEVDF - проблема спящих задач

Классической уязвимостью планировщиков на базе виртуального времени является обработка процессов, которые просыпаются после длительного сна (sleep)

В EEVDF это устранено механизмом <u>"отложенного исключения из очереди"</u> (diferred dequeue). Когда процесс уходит в сон, он фактически остается привязанным к виртуальной очереди (runqueue), но помечается специальным флагом

За это время сна ядро постепенно <u>"расщепляет"</u> (decay) его lag. К моменту реального пробуждения спустя часы, lag процесса математически сглаживается до приемлемых значений, исключая возможность эксплойта планировщика
<br>
<br>
<br>
### Метрика Load Average

**Load Average** - метрика, используемая для автомасштабирования серверов и оценки общей загрузки ОС <u>на основе очереди процессов на CPU</u>

Она выводится утилитами *uptime*, *top* и читается из */proc/loadavg* в виде трех чисел (экспоненциально взвешенных средних за 1, 5, 15 минут)

Значение 1.0 на одноядерной системе означает полную загрузку - значения <u>выше</u> указывают на <u>образование очередей</u>
<br>
<br>
<br>
### Специфика Load Average в Linux (Учет D-State)

В отличие от других Unix-систем, алгоритм calc_fold_active() в ядре Linux учитывает не только выполняющиеся задачи (nr_running), но также <u>плюсует к метрике D-state процессы</u>

В классическом Unix Load Average измерял только "спрос на CPU". Однако в LInux метрика была переосмыслена как <u>"спрос на системные ресурсы в целом"</u>

Если процесс заблокирован на чтении диска (состояние D), он не потребляет такты процессора, но система все равно напряжена - дисковая подсистема является бутылочным горлышком (bottleneck)
<br>
<br>
<br>
### Zombie процессы

**Z (Zombie)** процесс - процесс, <u>лишенный пользовательской памяти</u>, но <u>сохраняет крошечный объем ядерной памяти</u> (дескриптор, thread_info, стек ядра)

Главный исчерпываемый данными процессами ресурс - слоты в системной таблице процессов (PID)

Стек ядра (kernel stack) сохраняется для Зомби потому, что в нем могут находиться фрагменты статистики, необходимой для системных вызовов отчетности (например, getrusage).

Данные процессы возникают в случае <u>отсутствия кода завершения от потомка</u> при его завершении выполнения либо при <u>отсутствии запросов о статусе потомков</u> родителем (вызов wait())
<br>
<br>
<br>
### Orphan Processes

Ситуация усложняется, если родительский процесс завершается раньше, не успев вызвать wait() для своих потомков

Такие потомки классифицируются как "сироты" (**orphans**), так как их прямой родитель исчез из таблицы процессов, поэтому они не могут завершиться без системного вызова wait() от родителя

Для предотвращения утечки PID через вечных зомби используется механизм **re-parenting**, при котором процессу "сироте" назначается новый родитель (зачастую это PID 1), который бесконечно отправляет вызовы wait(), выступая "чистильтищиком" для всей ОС
<br>
<br>
<br>
### Subreapers

Переназначение всех сирот на PID 1 создавало проблемы для современных менеджеров процессов и контейнеризации. В ядре 3.4 был введен механизм **Subreaper** через системный вызов prctl(PR_SET_CHILD_SUBREAPER)

Паттерн двойного форка (double-forking) веками использовался демонами (например, Nginx) для отвязки от терминала. Родитель создает потомка и завершается, потомок становится сиротой, создает внука и тоже завершается. Внук - истинный демон

Система управления (systemd, supervisor) больше не знала PID этого внука и не могла получать уведомления о его смерти. Установив флаг Subreaper, процесс <u>начинает выступать в роли локального init для всего его дерева процессов</u>
<br>
<br>
<br>
### Подсистема сигналов

**Сигналы** в Linux - это <u>механизм программных прерываний</u> (Software interrupts) и асинхронного межпроцессорного взаимодействия (IPC - Inter-Process Communication)

Цель сигналов в Linux - <u>грубое, немедленное прерывание нормального потока выполнения процесса</u> (control flow)

Сигнал может быть сгенерирован ядром (при аппаратных исключениях - Hardware interrupts), отправлен другим процессом (kill()) или периферией (нажатие Ctrl + C)

Для каждого сигнала процесс может определить "**диспозицию**" (disposition): <u>проигнорировать, выполнить</u> действие по умолчанию или <u>перехватить</u> вызовом пользовательской функции-обработчика

Диспозиции обеспечивают гибкость, но безопасность ядра требует, чтобы сигналы SIGKILL и SIGSTOP было невозможно перехватить или проигнорировать
<br>
<br>
<br>
### Reliable Signals

Ядро Linux поддерживает <u>стандартные POSIX сигналы</u> (номера от 1 до 31), такие как SIGINT, SIGTERM, SIGCHLD. На уровне ядра ожидающие (pending) стандартные сигналы представлены не очередью, а битовой маской (bitmask)

Если сигнал определенного типа уже "висит" в ожидании (бит равен 1), все последующие сигналы этого же типа <u>безвозвратно отбрасываются</u>

Установка бита - атомарная ассемблерная операция BTS (Bit Test and Set), выполняющая за такт процессора и не требующая динамического выделения памяти в куче ядра. Потеря кратных сигналов (merging) приемлема для их семантики

Если пришло десять уведомлений SIGCHLD (ребенок умер), родителю достаточно проснуться один раз им запустить цикл waitpid(..., WNOHANG) до тех пор, пока не соберет всех мертвецов. Система проектировалась так, чтобы минимизировать накладные расходы на саму доставку
<br>
<br>
<br>
### Сигналы реального времени (POSIX RT Signals)

Для решения проблемы потери событий стандарт POSIX ввел "сигналы реального времени" (**Real-Time Signals**) , использующие номера от SIGRTMIN (32/64) до SIGRTMAX (64)

RT-сигналы в ядре поддерживают <u>полноценную очередизацию</u> (queuing): система сохраняет каждый экземпляр отправленного сигнала

В отличие от стандартных сигналов, RT-сигналы не имеют предопределенного системного значения и могут нести полезную нагрузку (sigval -  число или указатель)

По мере усложнения программного обеспечения разработчикам потребовался легкий механизм асинхронной передачи сообщений, не теряющий данные под нагрузкой. Архитектура RT-сигналов использует структуру очереди (с ограничением RLIMIT_SIGPENDING для защиты от DoS-атак на память ядра). Порядок их доставки строго детерминирован: сначала доставляются сигналы с меньшим номером (высокий приоритет), а сигналы одного типа доставляются строго в порядке FIFO
