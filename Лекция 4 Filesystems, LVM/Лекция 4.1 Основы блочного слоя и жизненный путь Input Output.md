#### Эволюция подсистемы ввода-вывода

Эпоха HDD - исторически ядро Linux использовало очередь (request_queue) для всех дисковых операций. Узким местом была механика магнитного диска

С появлением SSD и NVMe узким местом стала программная обработка и блокировки в самом процессоре

*blk-mq* - Был внедрен Multi-Queue Block IO - механизм, который распараллелил очереди и убрал глобальные блокировки для линейного масштабирования


#### Блочный слой ядра (Block Layer)

**Block Layer** - Это посредник между слоем Виртуальной Файловой системы (VFS) сверху и драйверами физ. устройств снизу

Позволяет ФС и приложениям не задумываться о том, работают ли они с локальным NVMe, аппаратным RAID или сетевым iSCSI

Блочный слой отвечает за буферизацию, объединение запросов, планирование (sheduling) и контроль лимитов устройств


#### Virtual File System

**VFS** - это подсистема ядра, представляющая собой слой абстракции, который позволяет клиентским приложениям прозрачно обращаться к различным ФС

VFS располагается между пользовательскими системными вызовами верхнего уровня и специфичными драйверами ФС нижнего уровня

Позволяет сосуществовать любому количеству различных ФС в рамках единого логического дерева каталогов (начиная от /)

VFS не хранит данные на диске и не передает их по сети. Это просто строгий контракт (интерфейс), который обязаны соблюдать все остальные

Интерфейс POSIX - VFS реализует стандартный интерфейс POSIX, оперирующий базовыми примитивами: open, read, write, close, seek

Virtual Methods - Базовые функции (open, read, write) определены в структуре file_operations как "виртуальные функции" без реализации по умолчанию. Конкретные ФС переопределяют их


#### Единая модель объектов (Common File Model)

Ядро сводит любую ФС (даже если у нее физически другая структура, как у FAT32) к единой объектной модели

Модель включает четыре главных сущности (метаданные):
- *Superblock* (Суперблок)
- *Inode* (Индексный дескриптор)
- *Dentry* (Элемент каталога)
- *File* (Открытый файл)

Драйвер FAT32 внутри яра вынужден "на лету" эмулировать inod'ы, чтобы угодить строгим правилам VFS


#### Superblock

Superblock представляет смонтированную ФС в целом

Хранит глобальную информацию: размер блока, максимальный размер файла, методы для аллокации инод, точку монтирования

VFS держит в памяти копии суперблоков всех активных ФС для мгновенного доступа к их глобальным настройкам

Если поврежден суперблок на диске - ФС не смонтируется. VFS просто не сможет инициализировать свой внутренний объект super_block


#### Inode (Index Node)

Представляет конкретный файл или директорию, но без его имени

Включает права доступа, владельца, временные метки и указатели на физ. блоки данных

VFS поддерживает Inode Cache в RAM, чтобы не читать эти метаданные с диска при каждом запросе

Например, когда делаем stat file.txt, VFS берет 90% информации именно из объекта inode


#### Dentry (Directory Entry)

Связующее звено. Объект dentry связывает строковое имя файла (например, "config.json") с конкретным номером Inode

Из объектов Dentry VFS строит древовидную структуру в RAM (Dentry Cache), отражающую иерархию папок

Благодаря Dentry Cache, разрешение пути /var/log/messages происходит за микросекунды в RAM

Dentry существует только в RAM как структура VFS. На диске нет физ. объекта "dentry"


#### Negative Dentries

Кэширование отсутствия - ядро кэширует не только существующие файлы, но и факт отсутствия файла на диске (Negative Dentries)

Повышение производительности - это ускоряет работу систем, которые часто ищут библиотеки в списке путей (например, LD_LIBRARY_PATH), возвращая ошибку без обращения к медленному диску

Negative dentries спасают дисковую подсистему при старте тяжелых приложений (Python/Java), которые проверяют десятки путей перед тем, как найти нужный .so или .jar файл


#### File (Открытый файл)

Представляет файл, открытый конкретным процессом. Создается в памяти в момент вызова open()

хранит текущее смещение (offset), флаги открытия (read-only, append) и ссылку на структуру Dentry

Два разных процесса могут открыть один и тот же файл. VFS создаст два объекта File (с разными смещениями), но они будут указывать на один inode.

Это объясняет, почему два логгера могут писать в один файл одновременно: у них разные File-объекты, но коллизии разрешаются на уровне общего inode


#### Виртуальные ФС в памяти

Прямой доступ к памяти - такие как *tmpfs*, *devtmpfs* и cgroups не реализуют функции file_operations для работы с диском, а взаимодействуют напрямую со структурами в памяти

VFS позволяет ядру экспортировать свои внутренние структуры (список процессов, прерываний, параметров) в виде читаемых текстовых файлов

*tmpfs* - это не просто "диск в оперативке", это ФС, которая напрямую интегрирована в механизм VFS Page Cache


#### Сетевые хранилища через VFS

Благодаря VFS приложение не замечает разницы между сохранением файла на локальный NVMe SSD и на сетевой NFS-сервер

Когда VFS видит, что Inode принадлежит NFS, он вместо вызова драйвера диска (Block Layer) отправляет системный вызов write() в сетевой стек (TCP/IP)

Например, контейнер пишет логи "в файл", не зная, что VFS пробрасывает эти данные на удаленный сервер хранения


#### Точка монтирования (Mount Points)

Точка монтирования - это путь внутри глобального дерева VFS, к которому привязывается корневой каталог другой ФС

Пользователь переходит из /var в /var/log и не замечает, что он физически пересек границу между двумя разными дисками с разными ФС

VFS поддерживает глобальную хэш-таблицу всех точек монтирования для быстрого разрешения путей при каждом обращении

Без VFS нам бы пришлось использовать буквенные диски (:D, :C) как в DOS/Windows, что разрушает концепцию единого дерева


#### Жизненный цикл IO запроса

- VFS инициирует системный вызов (read/write), формируя запрос к логическому файлу

- На уровне блочного слоя это намерение упаковывается в абстрактную структуру bio (block IO)

- После обработки и возможного слияния, bio превращается в аппаратную команду request, понятную драйверу

- Ошибки или задержки на уровне VFS это часто проблемы приложения. Задержки на уровне request - это проблемы диска или контроллера


#### bio - Базовая единица IO

bio описывает операцию блочного ввода-вывода (смещение на диске, размер, направление)

Объем данных, описываемый одной структурой bio, ограничен (например, максимум 4G)

Splitting - если bio превышает аппаратные лимиты контроллера, ядро "расщепляет" его цепочку мелких, связанных друг с другом bio

bio - это то, чего хочет приложение. Метрики в системных утилитах (*iostat*) часто показывают итоговые request, а не изначальные bio


#### request - Задание для драйвера

*request* содержит специфические для драйвера данные и таймеры контроля выполнения (например, *BLK_EH_RESET_TIMER*)

Один request может содержать внутри себя один или множество объединенных bio

Имеет строгий жизненный цикл (в очереди, в диспетчеризации, выполняется оборудованием)

Драйвер устройства (например, драйвер NVMe или SCSI) работает исключительно со структурами request, он ничего не знает про файлы или директории


#### Механизм Merging

Блочный слой пытается объединить новые bio с запросами, которые уже ожидают в очереди аппаратного драйвера

**Front-Merging** - Новые данные доклеиваются в начало существующего запроса (встречается редко)

**Back-Merging** - Новые данные доклеиваются в конец (типичный сценарий при последовательном чтении/записи)

Слияние превращает сотни крошечных операций по 4KB в один крупный блок на 512KB. Это радикально снижает накладные расходы контроллера диска

Слияние останавливается, когда достигаются лимиты контроллера (*max_hw_sectors*, *max_segment_size*)

Метрики iostat - Колонки rrqm/s (read requests merged) и wrqm/s показывают количество успешных слияний в секунду

Высокое значение rqm/s означает хорошую работу кэша и последовательный паттерн нагрузки. Значение "0" типично для Random IO или Direct IO

Посмотреть можно в *iostat -x 1*. Следует уметь сопоставлять физический профиль нагрузки с поведением блочного слоя


#### Plugging (Заглушки очередей)

Если отправлять каждый bio драйверу моментально, механизм слияния не успеет отработать (очередь всегда будет пустой)

VFS временно "затыкает" (plugs) очередь, позволяя накопиться пачке запросов от текущего процесса

*Unplug* - заглушка снимается явно (таймаут) или при переключении контекста (когда процесс засыпает, ожидая данных)


#### blk-mq

До *blk-mq* все ядра выстраивались в очередь к одной очереди (request_queue_lock) для доступа к диску

Структура данных очереди постоянно перемещалась между L1/L2 кэшами разных процессов, вызывая промахи и деградацию производительности

На системах с 32+ ядрами и NVMe процессоры тратили больше тактов на ожидание локов, чем на сам ввод-вывод

На графиках того времени можно было увидеть, как при добавлении новых ядер производительность диска парадоксальным образом падала из-за конкуренции блокировок


#### Архитектура Multi-Queue

*blk-mq* разбирает монолитную очередь на два слоя: программный (Software Queues) и аппаратный (Hardware Queues)

Процессы формируют IO локально на своем CPU, а оборудование забирает его параллельно из множества очередей

Это устраняет конкуренцию между ядрами CPU, позволяя достичь 15-20 миллионов IOPS на один сервер так как больше нет блокировок

Сегодня blk-mq включен по умолчанию для всех блочных устройств, даже для медленных USB-флешек


#### Программные очереди (Software Queues - ctx)

Per-CPU локальность - создаются индивидуально для каждого логического ядра процессора (структура *blk_mq_ctx*)

Процесс (например, Nginx), работающий на CPU #4, ставит bio только в программную очередь #4

На этом уровне происходит сортировка и предварительное слияние (merging) перед отправкой драйверу

Локальность данных (Date Locality) означает, что кэш процессора работает максимально эффективно, так как чужие ядра не изменяют структуры в памяти вашего ядра
